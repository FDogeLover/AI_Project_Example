# 进化机制触发时机

## 总览

```
机制              触发时机                      频率
─────────────    ──────────────────────       ─────
事前自检          每次开始工作前                每次
事后优化          每次完成一个完整任务后         每次
系统自省          完成 3 个以上任务后           每 3 次
知识保鲜          有新内容新增时                事件驱动
inbox 清理        inbox 超过 5 条时             事件驱动
跨项目发现        新项目接入时                  事件驱动
daily_report     每次对话结束时                每次
session_resume   新 session 开始时             事件驱动
```

**设计原则：有新内容才进化，没新内容不空转。**

## 触发条件详解

### 每次都做（每次对话）

**事前自检** — 在动手做任何事之前
```
触发：用户提出一个任务，你准备开始执行
动作：查 lab_book/mistakes/ + patterns/ + 项目 notes/
耗时：1-2 分钟
```

**事后优化** — 在完成一个完整任务后
```
触发：一个任务从头到尾做完了（不是中途暂停）
判断标准：
  - 用户确认"可以了" / "完成了"
  - 你交付了一个完整的结果
  - 一个功能点开发完毕
动作：
  1. 检查 mistakes/ → 新错误记录了吗？
  2. 检查 patterns/ → 新模式发现了吗？
  3. 检查 CONVENTIONS → 需要更新吗？
  4. 检查 skill_factory → 有值得做成 skill 的吗？
  5. 检查 tools/ → 有新工具值得复用吗？
耗时：2-3 分钟
```

**inbox 清理检查** — 每次对话结束时
```
触发：对话即将结束（用户说"先这样"、"下次再聊"）
动作：快速扫描 workspace/ 下的 inbox/，看有没有积压
  - 有积压 → 提醒用户
  - 没积压 → 跳过
耗时：30 秒
```

**daily_report** — 每次对话结束时
```
触发：同上
动作：自动运行 daily_report.py，记录今天的变更
耗时：10 秒
```

### 定期做（每 3 次任务后）

**系统自省** — 审视工作台本身
```
触发：累计完成 3 个以上任务后（不一定是同一次对话）
判断方法：数今天/近期的 lab_book/entries/ 新增了几条
动作：
  1. 这个工作台有没有不方便的地方？
  2. 有没有重复劳动可以自动化？
  3. 有没有知识散落各处应该整合？
耗时：5-10 分钟
```

### 按需做（有新内容时才触发）

**知识保鲜** — 检查知识库健康
```
触发：lab_book/ 或 workspace/ 下有新增内容时
判断方法：有新的 entries、patterns、mistakes 或项目变更
动作：运行 knowledge_guard.py
  - 过时的 pattern → 检查是否需要更新
  - 未解决的 mistakes → 检查是否已修复
  - 矛盾的 pattern → 检查并修正
耗时：10-15 分钟
```

**跨项目发现** — 新项目接入时
```
触发：workspace/ 下出现新目录
动作：运行 project_scanner.py，推荐可复用的经验
耗时：1 分钟
```

**session_resume** — 新 session 开始时
```
触发：新 session 开始，用户提到之前的工作
动作：运行 session_resume.py，快速恢复上下文
耗时：1 分钟
```

**inbox 清理** — inbox 有积压时
```
触发：inbox/ 下记录超过 5 条
动作：运行 inbox_cleaner.py，提示清理
耗时：30 秒
```

## 设计原则

```
有新内容 → 触发进化
没新内容 → 不做任何检查

进化是事件驱动的，不是时间驱动的。
不需要定时任务，有变更才行动。
